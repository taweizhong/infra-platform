# 模块 08：项目路线图与总任务清单

## 1. 里程碑规划

## M1：最小可用版本（4~6 周）

目标：配置中心 + KV 主链路打通。

- Hub：配置 CRUD、基础发布、Agent 推送。
- Agent：Config GET/Watch、KV CRUD/Watch。
- SDK：Config 初始化 + KV 基础 API。

验收：

- 单集群部署可稳定服务。
- 配置变更可在秒级推送到 SDK。
- KV 写入可在秒级被 Watch 感知。

## M2：高可用与稳定性（3~4 周）

目标：集群容错与性能优化。

- Leader 选举与切换。
- Redis Pub/Sub 多 Agent 协同。
- 冷启动预热与降级策略落地。

验收：

- Leader 故障后服务不中断。
- 10k 长轮询连接稳定运行。

## M3：治理能力（3~5 周）

目标：审批与发布流程完善。

- 飞书审批轮询。
- 生产多阶段发布状态机。
- 配置历史、Diff、回滚。

验收：

- 生产阶段可人工推进并支持回滚。
- 全流程有审计记录。

## M4：可选能力与生产强化（2~4 周）

目标：服务发现、SLO、故障演练。

- Discovery 可选能力上线。
- 全链路指标、日志、追踪完善。
- 压测与混沌演练。

## 3. 任务管理建议

1. 每个 Task 拆分为：设计、开发、测试、联调、验收子任务。
2. 所有任务必须绑定：负责人、预估工时、依赖任务、验收标准。
3. 每周输出：风险清单（技术债、瓶颈、阻塞）。

---

## 4. 风险与应对

- 风险 A：Leader 选型未定导致实现反复。
  - 应对：先抽象接口，使用 Redis Lock 快速落地，再替换实现。
- 风险 B：长轮询连接规模导致资源压力。
  - 应对：先做连接池与会话清理，再压测调参。
- 风险 C：加密/解密外部服务不稳定。
  - 应对：本地熔断 + 重试 + 明确失败策略。

---

## 5. Definition of Ready（DoR）

任务进入开发前必须满足：

- 接口协议已评审。
- 数据模型已评审。
- 依赖模块有明确 owner 与排期。
- 测试方案明确（单测/集成/压测）。

## 6. Definition of Done（DoD）

任务完成必须满足：

- 代码合入 + 文档更新。
- 自动化测试通过。
- 关键指标可观测。
- 回滚策略可执行。
